# Sprint Brief Guide

Two brief types are auto-generated by calendar sense:
- `impulse:sprint_brief` — fired 45 min before first sprint **review** (2+ reviews on same day)
- `impulse:sprint_planning_brief` — fired 45 min before first sprint **planning** session (2+ on same day)

## Deduplication Rule

**Never generate a sprint brief during an autonomous wake** — the calendar-sense automation handles timing via the impulse system. Generating proactively during an autonomous wake creates duplicates (confirmed incident: 2026-03-02, two planning briefs sent).

**Always check before generating**: If `state/notes/sprint-brief-YYYY-MM-DD.md` (for reviews) or `state/notes/sprint-brief-planning-YYYY-MM-DD.md` (for planning) already exists for today, skip — don't send a second brief.

Only generate a brief when explicitly triggered by `impulse:sprint_brief` or `impulse:sprint_planning_brief`.

## Sprint Review Brief

Triggered by `impulse:sprint_brief` when calendar sense detects 2+ sprint reviews on the same day.

## What to Produce

A consolidated summary per team of: Done, In Review/In Progress, and Won't Do items for the current sprint. Send as a single Discord message before the first review.

## Data Sources

| Team | Project | Filter |
|------|---------|--------|
| DA | Roadmap (#2) | `team_area=DA`, `sprint=<current>` |
| Nexus | Roadmap (#2) | `team_area=Nexus`, `sprint=<current>` |
| Nightshade | Roadmap (#2) | `team_area=Nightshade`, `sprint=<current>` |
| DevOps | Infrastructure (#9) | No sprint field — filter by `status` only |

## Determining Current Sprint

Query Roadmap project #2 for a few In Progress items with a known team:

```
github_project_items(project=2, status="In Progress", team_area="DA", max_items=3, verbose=true)
```

Look at the `Sprint` field in the results. The majority value is the current sprint (e.g., `"Sprint 16"`).

**Note:** Items may have mismatched sprint numbers (e.g., "Sprint 67" from a different numbering era). Use the most common sprint name, not the highest number.

## Query Pattern

For each team in the Roadmap project, run separate queries for each status:

```
github_project_items(project=2, team_area="DA", sprint="Sprint 16", status="Done")
github_project_items(project=2, team_area="DA", sprint="Sprint 16", status="In Review")
github_project_items(project=2, team_area="DA", sprint="Sprint 16", status="In Progress")
github_project_items(project=2, team_area="DA", sprint="Sprint 16", status="Won't Do")
```

For DevOps (project #9 — no sprint field):
```
github_project_items(project=9, status="In Progress")
github_project_items(project=9, status="Next up")
```

## Output Format

Send to Discord with `talk_to_user`. Include:
- Header: "Sprint Brief — [Date] — [N] reviews today: [Team list]"
- One section per team with review start time
- Bullets by status category (Done / In Review / In Progress / Won't Do)
- Link issues where possible
- Keep it scannable — no paragraph prose

## Example Section

```
**DA Sprint Review — 11:30**
*Sprint 16 | Roadmap project*

Done:
- Item title — #123
- Item title — #456

In Progress:
- Item title — #789

Won't Do:
- Item title — #101
```

## Edge Cases

- **Only one sprint review**: Brief is NOT triggered (threshold: 2+ reviews on same day). Use `/sprint-brief` on-demand if needed.
- **Nightshade missing data**: Nightshade tracks in Roadmap #2 with `team_area=Nightshade`. If items are missing, note it in the brief.
- **DevOps sprint field**: Infrastructure project (#9) has no sprint field. Query by status only.
- **Sprint name ambiguity**: If sprint names are inconsistent (mixed Sprint 16, Sprint 67), use the name that appears on the most recent In Progress items.
- **Multi-era Not Completed noise**: Some teams (e.g., Nexus) have "Not Completed" items from old sprint numbering eras (Sprint 69+). Always filter by sprint name — querying `status=Not Completed` without a sprint filter surfaces old backlog items that look like active carryovers.
- **Items without sprint field**: Some "In Progress" items have no Sprint field assigned — they're ongoing/continuous work not tied to a sprint commitment. If they appear alongside sprint-filtered results, note them separately as "ongoing work (no sprint assigned)" rather than treating them as sprint items.

## After Generating

Save a copy to `state/notes/sprint-brief-YYYY-MM-DD.md` for reference.

## Sprint Planning Brief

Triggered by `impulse:sprint_planning_brief` when calendar sense detects 2+ sprint planning sessions on the same day.

### What to Produce

A consolidated pre-planning summary per team showing what carried over from the previous sprint. Teams will define the new sprint during the sessions — this brief helps them walk in prepared.

### Query Pattern

Planning briefs query the **previous sprint's** items to surface what needs attention at Monday's meeting. The new sprint's items are added during the meeting itself, so they won't be in GitHub yet.

**Detect the previous sprint name** by looking at Not Completed or Done items (In Progress items on Monday morning are edge cases — the sprint may not be fully closed):

```
# Find previous sprint by looking at recently closed items:
github_project_items(project=2, status="Not Completed", team_area="DA", max_items=3, verbose=true)
github_project_items(project=2, status="Done", team_area="DA", max_items=3, verbose=true)

# For each team, query previous sprint items:
github_project_items(project=2, team_area="DA", sprint="Sprint 16", status="Not Completed")
github_project_items(project=2, team_area="DA", sprint="Sprint 16", status="Done")  # for context
# Skip "Won't Do" — intentionally dropped, not relevant
# Skip "In Progress" — should not exist at sprint end; any remaining are edge cases (still open, not yet marked)

# Backlog candidates (no sprint assigned) — teams pick from these during planning:
github_project_items(project=2, team_area="DA", sprint="backlog")
github_project_items(project=2, team_area="Nexus", sprint="backlog")
github_project_items(project=2, team_area="Nightshade", sprint="backlog")
# Filter out "Won't Do" items from backlog results — they're intentionally dropped

# DevOps (no sprint field):
github_project_items(project=9, status="In Progress")
github_project_items(project=9, status="Next up")
```

**Key distinction:** "Not Completed" ≠ carryover. It means the task failed to finish. At Monday's meeting, the team decides which get **new tasks** in the current sprint. The old "Not Completed" item stays closed — it is never moved forward.

**Edge case:** On Monday morning before finalization, some items from last sprint may still be open (In Progress). Note these separately as "Still open from last sprint — not yet finalized."

**Filtering note:** When querying `status=Not Completed` without a sprint filter, items from old sprint numbering eras may appear (e.g., Sprint 69+ for Nexus before the numbering reset to Sprint 16). Always include `sprint="Sprint N"` to avoid surfacing stale backlog items as current carryovers.

### Output Format

Send to Discord with `talk_to_user`. Include:
- Header: "Sprint Planning Brief — [Date] — [N] sessions today: [Team list]"
- One section per team with planning session time
- Sections: "Not Completed last sprint (retry decisions)", "Done last sprint", "Backlog candidates", then optionally "Still open (pre-finalization)"
- Omit "Won't Do" items from backlog candidates
- Note that the new sprint's items won't be in GitHub yet — teams add them during planning
- Keep it scannable

### Example Section

```
**DA Sprint Planning — 11:30**
*Sprint 16 wrap-up | Roadmap project*

Not Completed (retry decisions needed):
- Infinity: make header more concise
- Test to determine network size for testnet

Done last sprint:
- Get verification of check auth working ✅
- Fix codeword merkle root check bug ✅

Backlog candidates (no sprint assigned):
- Update FRIvail paper [Draft]
- Infinity: Spec out light node protocol [Draft]

Still open from last sprint (not yet finalized):
- Deployment plan (monitoring, signoz dashboard, etc)
```

### After Generating

Save a copy to `state/notes/sprint-brief-planning-YYYY-MM-DD.md` for reference.
